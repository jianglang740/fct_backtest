
# Parquet 文件格式详解 + 和 CSV/原生DataFrame对比 + 私募量化回测为什么用它

## 一、Parquet 基础核心知识点

### 1. 本质是什么

**Parquet 是列式存储（Columnar Storage）的二进制文件格式**，由Apache开源，专门为大数据、结构化表格数据设计，不是文本文件（CSV是纯文本）,它的优势是程序将数据从硬盘读取到内存当中是极快，且其本身在内存中的占用空间较小。


你可以简单理解：

- CSV：**按行存数据**，一行就是一条股票K线记录；
- Parquet：**按列存数据**，单独存一整列`trade_date`、一整列`close`、一整列因子`fct_1`。

### 2. 两大核心底层特性

#### （1）列式存储

举个你回测场景的例子：
你的表字段：`trade_date,code,open,high,low,close,fct_1,fct_2`

- 如果你只需要**日期+因子fct_1**做单因子分层，CSV必须把整行全部读入内存再删列；
- Parquet 可以**直接只读取指定两列**，完全不加载open/high/low这些无用列，IO和内存占用直接大幅下降。

#### （2）内置压缩 + 分块索引

文件自带多种压缩算法（Snappy、GZIP、Brotli），而且会对数据分块、加统计索引（列最大值/最小值）：
比如你筛选`trade_date >= 2023-01-01`，Parquet会直接跳过不在时间范围内的数据块，不用遍历全表逐行判断。

### 3. Pandas 读写基础语法（你代码里就能直接用）

- Pandas 读取 parquet 时，会**把分散存储的多列重新拼接成二维表格 DataFrame，但磁盘文件层面，它从来不是一行一条记录**

```python
# 读取parquet，和read_csv几乎一样简单
df = pd.read_parquet("price_data.parquet")

# 只读取需要的列，回测最常用优化
df = pd.read_parquet("price_data.parquet", columns=["trade_date","code","close","fct_1"])

# 写入parquet
df.to_parquet("new_data.parquet", compression="snappy")
```

### 4. parquet的优势

优势 **只体现在【从硬盘往内存读数据】这一步** ，一旦数据进入内存变成 DataFrame，列式特性就消失了：

1. 只想用 2 列因子做回测 → 只读取硬盘里这两列，其余几十列完全不加载，省硬盘 IO、省内存；
2. 想按日期筛选数据 → 读取文件时直接根据列索引跳过不需要的数据块，不用全读进来再删行。

等数据加载进内存之后，你想怎么改行、删行、插行、打乱行顺序，Pandas 全都可以正常执行。

后续如果需要把修改后的数据落盘保存，`df.to_parquet()` 会再次自动按列式压缩存入磁盘。


## 二、Parquet VS CSV VS 直接内存DataFrame 全方位对比

### 1. 文件体积（磁盘占用）

| 格式          | 大小说明                                                                | 量化数据场景差距                                                  |
| ------------- | ----------------------------------------------------------------------- | ----------------------------------------------------------------- |
| CSV           | 纯明文文本，数字、日期全用字符串存储，无压缩，体积最大                  | 同一份5年全市场股票K线数据，CSV通常是Parquet**3~10倍** 大小 |
| Parquet       | 列式存储+列级压缩，重复值（比如股票代码、交易日期大量重复）压缩效率极高 | 磁盘占用极小，硬盘存海量历史因子库非常划算                        |
| 内存DataFrame | 完全放在RAM里，不落地文件；程序关闭数据消失                             | 无法持久保存历史数据源，每次重启必须重跑生成                      |

### 2. 读取速度（回测最关键）

1. **CSV**：必须整文件逐行解析文本转数值，哪怕只需要1列也要读完整个文件，海量数据（全A股多年日线）读取极慢；
2. **Parquet**：
   - 按需列裁剪：只读你回测用到的字段；
   - 谓词下推：读取时直接按日期/股票代码过滤，不用全量载入后再`df[df.trade_date>xxx]`；
     大数据集下读取速度通常是CSV **5~20倍**；
3. **内存DataFrame**：读取最快，但前提是数据已经在内存里，数据源不能断电保存，每次启动脚本要重新加载原始文件。

### 3. 数据类型兼容性（量化避坑重点）

- CSV：**丢失原生数据类型**，所有内容默认读成字符串，必须手动指定`dtype`，很容易出现股票代码`000001`变成数字丢失前导0、日期解析出错、浮点数精度异常；
- Parquet：**文件内部自带每一列的数据类型元信息**，int64、datetime64、float64、category分类类型会永久保存在文件里，pandas读取后不用手动转换类型，不会出现格式错乱，非常适合标准化因子库。

### 4. 分块与大数据支持

当数据量超级大（比如全A股10年日线+上百个因子，几百万行）：

- CSV分块`chunksize`只能按行切分，逻辑麻烦；
- Parquet原生支持**分区存储**（比如按年份/月份分成文件夹分区），你可以把`price_data`按年份分文件夹，回测2024年数据直接只读2024分区，完全不碰其他年份数据，这是私募因子库最常用的管理方式。

### 5. 缺点（什么时候还用CSV）

Parquet**不能用记事本/WPS直接打开查看**，必须用pandas、Spark、DuckDB等工具读取；
CSV优势就是随手用Excel打开看几行小样数据，适合临时小样本调试，不适合做长期批量回测数据源。

## 三、你们私募回测框架强制用Parquet的核心原因（贴合你的实习场景）

### 1. 公司会维护海量标准化因子库，节约服务器硬盘

私募会同时维护几十上百个单因子（fct_1~fct_n）、多年份全市场股票数据，每天增量更新K线与因子。
如果全部存CSV，服务器硬盘会被海量文本文件占满；Parquet高压缩特性可以把历史因子数据库体积压到极小，方便长期归档。

### 2. 单因子批量回测极致提速，节约研发时间

你们框架是单因子批量遍历回测：会轮流读取不同因子列做分组收益、IC、分层净值计算。
用CSV每次回测都要全表加载所有无用列；
Parquet可以**每次只读取当前正在测试的那一列因子+基础行情列**，内存占用直接砍80%以上，批量跑几十个因子回测效率天差地别。

### 3. 避免数据格式脏数据，统一团队数据口径

量化团队最忌讳：A同事导出CSV代码是浮点数，B同事读的时候转int出错；日期有的存字符串有的存datetime。
Parquet固化了每一列的数据类型，整个团队读取同一份parquet数据源，不会出现因读写方式不同导致的数据bug，合规且方便投研复盘。

### 4. 方便对接大数据集群（私募投研基建标配）

中型私募一般会用Spark、DuckDB、ClickHouse做底层数据仓库，这些大数据引擎**原生优先支持Parquet**，后续如果需要把回测框架上云、接入数仓，parquet文件可以无缝迁移，CSV需要额外做ETL转换。

### 5. 支持增量更新数据源

每日收盘后只需要追加当日行情与因子，直接往parquet数据集追加写入即可；CSV追加容易出现换行、编码错乱、表头重复等问题，维护自动化每日数据链路麻烦很多。

## 四、给你实习上手的2个实用小建议

1. **日常查看小样数据**
   不要直接`pd.read_parquet(大文件)`全量读取，用：

```python
# 只读前10行+指定列，不会加载整个几百万行表
df = pd.read_parquet("price_data.parquet", columns=["trade_date","code","close","fct_1"]).head(10)
print(df)
```

避免一次性加载超大DataFrame把本机内存占满卡顿。

2. **自己新构造因子后保存**
   生成新因子列后，优先保存为parquet而不是csv：

```python
df.to_parquet("new_factor_dataset.parquet", compression="snappy", index=False)
```

Snappy压缩是速度和体积平衡最好的选择，也是行业默认。

## 极简总结

CSV = 方便肉眼查看的**文本临时文件**；
Parquet = 面向量化大数据回测、**列式压缩、按需读取、类型固化、适合团队长期维护因子数据库**的工业级二进制存储格式，也是国内绝大多数量化私募、券商金工组回测系统的标准底层数据源。
原生内存DataFrame只是程序运行时临时数据，无法落地持久化做数据源。
